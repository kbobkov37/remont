ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Наименование проекта: Веб-приложение для управления сервисным центром по ремонту электроники (ServiceDesk Repair WebApp)  
Версия ТЗ: 1.1  
Дата: 26.05.2026  
Изменения: Удалены платёжные интеграции; добавлен модуль «Сторонние подрядчики».
1. Общие сведения
Параметр	Описание
Цель	Автоматизация приёма, диагностики, ремонта (собственного и аутсорс), учёта запчастей, расчётов с клиентами и контрагентами, аналитики работы СЦ.
Целевая аудитория	Клиенты (B2C/B2B), мастера-инженеры, менеджеры, руководители СЦ, внешние подрядчики (опционально — доступ к порталу).
Платформа	Веб-приложение (PWA), адаптивное под desktop/tablet/mobile.
Язык интерфейса	Русский (возможность мультиязычности в v2).
Финансы	Без онлайн-оплаты в MVP. Расчёты с клиентами — офлайн (наличные/терминал/перевод), фиксация факта оплаты в системе.
2. Ролевая модель и функциональные требования
2.1 Клиент
Регистрация/авторизация (email, телефон, OAuth: Google/Яндекс)
Создание заявки: выбор категории → бренда → модели → описание неисправности → загрузка фото/видео
Выбор услуги: диагностика, срочный ремонт, выезд мастера
Отслеживание статуса в реальном времени (включая статус «Передано подрядчику»)
Фиксация оплаты: отметка «Оплачено» (менеджером), просмотр истории платежей, получение чека (загружается вручную или импортируется из учётной системы)
История заявок, акты, гарантийные талоны
Чат/комментарии с менеджером, оценка качества (NPS/звёзды)
Push/SMS/email уведомления о смене статуса
2.2 Мастер-инженер (внутренний)
Личный кабинет с назначенными заявками
Изменение статусов: `Принят → Диагностика → Ожидание запчастей → В ремонте → Тестирование → Готов к выдаче → Выдан`
Инициирование передачи заявки подрядчику (запрос на аутсорс при отсутствии компетенций/запчастей)
Внесение результатов диагностики, добавление работ и запчастей из справочника
Фотофиксация этапов, заметки, внутренние комментарии
Учёт затраченного времени, формирование акта выполненных работ
2.3 Администратор / Менеджер
Дашборд: загрузка СЦ, просроченные заявки, выручка, маржинальность (в т.ч. по аутсорсу)
Управление заявками: фильтрация, приоритизация, назначение (внутреннему мастеру / подрядчику)
Модуль «Подрядчики»:
Справочник контрагентов: наименование, ИНН/ОГРН, контакты, специализация, рейтинг, условия сотрудничества
Создание заявки подрядчику: выбор контрагента, автоматическая генерация акта передачи (PDF), отправка по email/в личный кабинет
Отслеживание статуса у подрядчика: `Передан → В работе у подрядчика → Готов у подрядчика → Возвращён в СЦ`
Учёт себестоимости работ подрядчика, расчёт маржи (цена для клиента − стоимость аутсорса)
История взаимодействий, документы, лимиты и дебиторская задолженность
Справочники: устройства, модели, неисправности, прайс-лист, склад запчастей
Генерация документов: договор приёма, акт диагностики, счёт, гарантийный талон (PDF)
Управление пользователями и ролями, лог действий
Настройка шаблонов уведомлений, SLA-правила
2.4 Руководитель
Финансовая аналитика: маржинальность по мастерам/услугам/подрядчикам, ABC-анализ запчастей
KPI: среднее время ремонта (собственный/аутсорс), % повторных обращений, загрузка мощности
Отчёты для бухгалтерии: взаиморасчёты с контрагентами, акты сверки

5.1 Статусная машина заявки
Таблица статусов и переходов
Статус	Описание	Кто может изменить	Допустимые переходы
Created	Заявка создана, ожидает подтверждения	Клиент	→ `Confirmed`, → `Rejected`
Confirmed	Заявка подтверждена менеджером, принята в работу	Менеджер	→ `Diagnostic`, → `OutsourcingRequested`
Rejected	Заявка отклонена (неполные данные, отказ клиента)	Менеджер, Клиент	→ `Created` (пересоздание)
Diagnostic	Проводится диагностика устройства	Мастер	→ `InProgress`, → `WaitingParts`, → `OutsourcingRequested`, → `Completed` (если диагностика — итоговая услуга)
WaitingParts	Ожидание поступления запчастей	Менеджер, Мастер	→ `InProgress`, → `Cancelled` (при отказе клиента)
InProgress	Устройство в ремонте	Мастер	→ `Testing`, → `WaitingParts`, → `OutsourcingRequested`
Testing	Тестирование после ремонта	Мастер	→ `ReadyForPickup`, → `InProgress` (требует доработки)
ReadyForPickup	Ремонт завершён, готов к выдаче	Мастер	→ `Completed`, → `Cancelled` (не вывезено >30 дней)
Completed	Устройство выдано клиенту, ремонт завершён	Менеджер	— (финальный статус)
Cancelled	Заявка отменена	Менеджер, Клиент	→ `Created` (по желанию клиента)
OutsourcingRequested	Запрос на передачу подрядчику (внутренний статус)	Мастер, Менеджер	→ `Outsourced`, → `InProgress` (отмена передачи)
Outsourced	Устройство передано подрядчику	Менеджер	→ `OutsourcedDone`, → `OutsourcedRejected`, → `Outsourced` (с подстатусами)
Подстатусы аутсорса (когда `Outsourced`)
Подстатус	Описание	Кто меняет	Переходы
`Outsourced.InTransit`	Устройство в пути к подрядчику	Менеджер	→ `Outsourced.InWork`
`Outsourced.InWork`	Подрядчик выполняет работу	Подрядчик (через портал) / Менеджер	→ `Outsourced.QualityCheck`, → `Outsourced.Rejected`
`Outsourced.QualityCheck`	Проверка качества у подрядчика	Подрядчик	→ `Outsourced.Done`, → `Outsourced.InWork`
`Outsourced.Done`	Работа у подрядчика завершена, готов возврат	Подрядчик / Менеджер	→ `InTransitBack`
`Outsourced.Rejected`	Подрядчик отказался от выполнения	Подрядчик	→ `InProgress` (возврат в СЦ), → `OutsourcingRequested` (другой подрядчик)
`InTransitBack`	Устройство возвращается в СЦ	Менеджер	→ `Outsourced.Done` (получено)
Диаграмма переходов (текстовое представление)
```
┌─────────────┐
│  Created    │
└──────┬──────┘
       │Confirmed
       ▼
┌─────────────┐
│ Confirmed   │
└──────┬──────┘
       │Diagnostic / OutsourcingRequested
       ▼
┌───────────────┐     ┌─────────────────────┐
│  Diagnostic   │────▶│ OutsourcingRequested│
└───────┬───────┘     └──────────┬──────────┘
        │                        │Outsourced
        ▼                        ▼
┌───────────────┐     ┌─────────────────────┐
│ WaitingParts  │     │    Outsourced*      │
└───────┬───────┘     │ (см. подстатусы)    │
        │InProgress   └──────────┬──────────┘
        ▼                        │Возврат в СЦ
┌───────────────┐                ▼
│   InProgress  │◀────────────────┘
└───────┬───────┘
        │Testing
        ▼
┌───────────────┐
│   Testing     │
└───────┬───────┘
        │ReadyForPickup
        ▼
┌───────────────────┐
│ ReadyForPickup    │
└───────┬───────────┘
        │Completed
        ▼
┌───────────────┐
│   Completed   │◀──────────────────────┐
└───────────────┘                       │
        ▲                               │Cancelled
        │Cancelled                      │
┌───────┴───────┐                       │
│  WaitingParts │───────────────────────┘
└───────────────┘
```
Правила бизнес-логики статусов
Правило	Описание
SLA-таймеры	Для каждого статуса (кроме `WaitingParts`) вычисляется время нахождения в статусе. Превышение SLA → автоматическая эскалация менеджеру
Одновременные переходы	Запрещено: один переход может выполнить только один пользователь за раз (оптимистическая блокировка по `version`)
Аудит переходов	Каждый переход логируется: `user_id`, `timestamp`, `from_status`, `to_status`, `comment` (опционально)
Уведомления	При переходе в `ReadyForPickup`, `Completed`, `Cancelled` — уведомление клиента
Возврат из аутсорса	При возврате устройства от подрядчика — автоматический переход в `InProgress` с комментарием о причине возврата
Финальные статусы	`Completed`, `Cancelled` — блокируют изменение статуса без роли `Admin`
Время в аутсорсе	Отдельный SLA для аутсорса: суммарное время всех подстатусов `Outsourced.*` не должно превышать X дней

7. Требования к тестированию и приёмке
Покрытие unit-тестами ≥ 70%, E2E-сценарии по ключевым ролям
Сценарии передачи заявки подрядчику и возврата в СЦ (включая отмену передачи)
Проверка расчёта маржинальности при аутсорсе
Валидация генерации актов приёма-передачи (корректные реквизиты, подписи)
Тесты на разграничение прав: подрядчик не видит данные других клиентов
Нагрузочное тестирование (k6/Locust) до целевой нагрузки ×1.5
Security-аудит (OWASP Top 10, Dependabot/Snyk)
UAT с участием 2-3 реальных мастеров и менеджеров
Критерий приёмки: все пункты раздела 2 реализованы, багов критического/высокого приоритета = 0, документация передана
8. Поддержка, развитие и риски
SLA поддержки: 24/5 мониторинг, реакция на критические инциденты ≤ 2 ч
Бэкапы: ежедневные инкрементальные, еженедельные полные, хранение ≥ 30 дней
Документация: пользовательская (PDF/Notion), API (Swagger), развёртывание (README)
Риски и mitigations:
Задержки со стороны подрядчиков → SLA-контроль и авто-эскалация
Расхождения в качестве → система рейтингов и блокировка ненадёжных контрагентов
Юридические нюансы передачи техники → шаблоны договоров, обязательная фотофиксация при передаче/приёмке
Roadmap v2:
Личный кабинет для подрядчиков (упрощённый доступ)
Интеграция с сервисами доставки для логистики к подрядчику
AI-рекомендации: какому подрядчику назначить заявку на основе специализации и загрузки
Интеграция с 1С/облачной бухгалтерией
9. Юридические и compliance требования
Обработка ПДн в соответствии с 152-ФЗ (политика конфиденциальности, согласие, хранение в РФ при необходимости)
Договоры с контрагентами: шаблоны в системе, учёт версий, напоминания о продлении
Акт приёма-передачи техники: юридически значимый шаблон (с возможностью ЭЦП при необходимости)
При передаче данных подрядчику — фиксация согласия клиента (галочка в форме приёма)
Учёт гарантийных обязательств: разграничение гарантии «от СЦ» и «от подрядчика»
Логирование действий для аудита и разрешения споров
10. Критерии завершения проекта
Развёрнутая production-среда с доменом, SSL, мониторингом
Реализован и протестирован полный цикл: приём → диагностика → передача подрядчику → возврат → выдача клиенту
Передача исходного кода, доступа к репозиториям, учётных данных
Подписанный акт сдачи-приёмки работ
Проведённое обучение персонала (не менее 2 сессий, включая модуль «Подрядчики»)
Предоставлена гарантия на исправление багов (обычно 3-6 месяцев)

🔧 Рекомендации по реализации модуля аутсорса
Статусная машина: Используйте явный статус `Outsourced` с подстатусами, чтобы не смешивать внутренний и внешний ремонт.
Документооборот: Генерируйте акт приёма-передачи в формате PDF с уникальным номером и QR-кодом для быстрой сверки при возврате техники.
Финансы: Храните два значения в заявке — `price_for_client` и `contractor_cost`, чтобы автоматически считать маржу.
Рейтинг подрядчиков: Внедрите систему оценки (1-5) после возврата техники → влияет на авто-подбор и видимость в дашборде.
Безопасность: Ограничьте видимость персональных данных клиента для подрядчика (только модель устройства и неисправность, если это допустимо по договору).
